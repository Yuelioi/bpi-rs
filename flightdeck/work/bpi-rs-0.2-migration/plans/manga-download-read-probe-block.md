# Manga Download Read Probe Block

## Scope

`manga/download-read` covers the unimplemented current reader endpoints formerly attempted
from `src/manga/download.rs`:

- `POST https://manga.bilibili.com/twirp/comic.v1.Comic/GetImageIndex`
- `POST https://manga.bilibili.com/twirp/comic.v1.Comic/ImageToken`

`ImageToken` depends on image paths returned by `GetImageIndex`, so `GetImageIndex` must
succeed first.

## Probe Attempts

Local raw Probe outputs stay under `target/bpi-probe-runs/manga/download-read/...`.

Attempts made so far:

- Existing trials for `ep_id = 482133` with anonymous, normal, and vip profiles.
- Existing web-style trials for `ep_id = 321912` with `device=pc&platform=web`, manga
  referer, origin, and browser-like headers.
- Existing anonymous `ComicDetail` discovery trials for `comic_id = 24442`.
- Current normal/vip `ComicDetail` discovery trials for `comic_id = 24442`, using
  `device=pc&platform=web`, cookie, manga referer, origin, and browser-like headers.
- Current normal/vip `GetImageIndex` trials using camelCase `epId = 321912`.
- Current normal/vip `GetImageIndex` trials using another public README sample
  `ep_id = 334263` with `mc26742/334263` referer.
- User-provided current reader pages were tested:
  - free preview: `https://manga.bilibili.com/mc28218/469901?from=manga_detail`
  - paid/VIP-owned: `https://manga.bilibili.com/mc28218/469956`

All current and prior `GetImageIndex` / `ComicDetail` download-read probes returned HTTP
200 with API `code = 99` when using the old SDK request shape. Authenticated profiles
returned the generic business-exception message. No promoted success contract should be
created from those old-shape attempts.

The user-provided reader pages prove that the chapter ids and VIP entitlement are valid,
but also prove the current web contract is no longer the old SDK shape:

```text
CDP real-browser capture for free `mc28218/469901`:
  GetImageIndex request body keys: ep_id, m2
  GetImageIndex response: HTTP 200, API code 0, 50 images, host present
  ImageToken request body keys: m1, urls
  ImageToken response: HTTP 200, API code 0, one token per request

CDP real-browser capture for paid/VIP-owned `mc28218/469956`:
  GetImageIndex request body keys: ep_id, m2
  GetImageIndex response: HTTP 200, API code 0, 48 images, host present
  ImageToken request body keys: m1, urls
  ImageToken response: HTTP 200, API code 0, one token per request
```

The `m1`/`m2` values were generated by the current reader frontend and were not printed,
committed, or promoted. Local summaries are under
`target/bpi-probe-runs/manga/download-read/provided-links/...` and
`target/manga-cdp-*-capture-summary.json`.

Additional Playwright research used the same local VIP account cookies without printing or
committing secrets/proof values. Local-only scripts and sanitized summaries are under
`target/playwright-research/`, `target/manga-playwright-research.json`, and
`target/manga-playwright-proof-hook.json`.

Playwright confirmed the current reader path:

```text
free `mc28218/469901`:
  GetImageIndex: body keys ep_id,m2; API code 0; 50 images
  ImageToken: body keys m1,urls; API code 0; one token per request

paid/VIP-owned `mc28218/469956`:
  GetImageIndex: body keys ep_id,m2; API code 0; 48 images
  ImageToken: body keys m1,urls; API code 0; one token per request
```

The reader JS loads these WASM assets:

```text
mCmb -> 36aafde75cfc07c30970.wasm -> Go WASM exposing y1_z2w2a3 through syscall/js
kb0z -> 7f0cd9bae030a8363604.wasm -> Go WASM exposing a1_o8iso5 through syscall/js
aHO9 -> dda35c98742815151e46.wasm -> Go WASM exposing a1_h17mj9 through syscall/js
vY5y -> ca09620080ee3723f9c8.wasm -> Go WASM for devtools/reporting state
uIl/ -> 28d51de1197fc9acdfe4.module.wasm -> non-Go wasm_utils helper
```

The hook run shows `a1_o8iso5` returns the value used as `m2`; for the current chapter
request, its input is a non-decimal 14-byte string containing the episode id, and its
output length matches the `m2` request-body length. `y1_z2w2a3` signs JSON-shaped payload
strings and returns an object with a 48-byte `sign` field. `ImageToken` sends an 88-byte
`m1` field plus stringified `urls`; that field is generated by the reader wrapper path, not
by any stable legacy SDK parameter. `a1_h17mj9` is also invoked by the reader to decode
image payload data.

These modules are Go WASM runtimes (`run`, `resume`, `getsp`, `mem` exports plus `gojs.*`
imports), not simple standalone exported functions. A pure Rust implementation would need
to either port the browser/Go-WASM runtime behavior or define an explicit proof-provider
boundary.

## Reference Check

External references still describe the same PC request shape already tested here:

- `GetImageIndex?device=pc&platform=web`
- Cookie-backed request
- JSON body containing `ep_id`
- manga chapter referer and browser-style headers

Those references support the request-shape trial, but local Probe remains authoritative for
this workspace and currently contradicts promotion. The current reader JS additionally sends
`m2` with `GetImageIndex` and `m1` with `ImageToken`, generated from frontend WASM/browser
state.

## Decision

Mark `manga/download-read` as not implemented for the current SDK migration. The old
private helper implementation has been removed from `src/manga/download.rs`; the module now
keeps response models only and documents the missing proof-provider boundary.

Reopen only if one of these is explicitly selected as a dedicated batch:

- an explicit low-level API design where callers provide `m2` for `GetImageIndex` and `m1`
  for `ImageToken`;
- or a dedicated reader proof provider backed by browser/Playwright or a maintained
  Go-WASM execution path;
- or a researched pure-Rust proof implementation, accepted as a separate high-risk batch.

Do not promote old-shape `GetImageIndex` or `ImageToken` contracts from `code = 99`
responses. The readable chapter evidence exists, but default migration work should continue
elsewhere instead of spending more time on this feature.
